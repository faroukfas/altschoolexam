PROJECT OVERVIEW

This project involved building and deploying a fully production-grade static website on AWS, hosted entirely within the free tier. What started as a simple static site hosting exercise evolved into a complete Infrastructure as Code (IaC) and CI/CD pipeline project — covering everything from DNS and SSL certificates to automated deployments triggered by a Git push.

The live site is accessible at https://resq.com.ng and is served globally over HTTPS with a custom domain, managed SSL certificate, and CloudFront CDN caching.

ARCHITECTURE

The infrastructure is built across four AWS services that work together in a chain:

1

Amazon S3

Stores all static website files — HTML, CSS, and images. The bucket is kept private and only accessible by CloudFront via Origin Access Control (OAC).

✓ 5 GB storage free · 20K GET / 2K PUT requests free

2

AWS Certificate Manager

Issues a free SSL/TLS certificate for resq.com.ng and *.resq.com.ng. Auto-renews as long as the DNS validation record stays in Route 53.

✓ Free SSL certificate with CloudFront

3

Amazon CloudFront

Serves the website globally via AWS edge locations. Enforces HTTPS, caches files for performance, and acts as the secure entry point in front of S3.

✓ 50 GB transfer free · 2M requests free/month

4

Amazon Route 53

Manages DNS for resq.com.ng. Alias A records point the root domain and www subdomain directly to the CloudFront distribution.

✓ Free DNS queries for A records

Traffic Flow

Browser → Route 53 → CloudFront + ACM → S3 Origin

TOOLS & TECHNOLOGIES

Beyond the AWS services, two key tools were used to manage infrastructure and automate deployments:

Terraform

All AWS infrastructure is written as code using Terraform. This means the entire setup — S3, CloudFront, Route 53, ACM, and IAM — can be reproduced, version controlled, and modified through code rather than clicking through the AWS console.

Existing resources (S3 and CloudFront) were imported into Terraform state so they could be managed consistently alongside the new resources.

GitHub Actions

A CI/CD pipeline was set up using GitHub Actions. Every time code is pushed to the main branch, the workflow automatically syncs the updated files to S3 and creates a CloudFront cache invalidation — meaning the live site updates within minutes of a Git push.

A dedicated IAM user with least-privilege permissions was created specifically for the pipeline, with credentials stored securely as GitHub Secrets.

THE CI/CD PIPELINE

Here is exactly what happens every time code is pushed to the main branch on GitHub:

1
Code pushed to GitHub

A git push to the main branch triggers the GitHub Actions workflow automatically.

2
AWS credentials configured

GitHub Actions uses stored secrets to authenticate with AWS as the dedicated deployer IAM user.

3
Files synced to S3

The workflow runs aws s3 sync, uploading changed files and removing deleted ones from the bucket.

4
CloudFront cache invalidated

A /* invalidation is created so CloudFront fetches fresh files from S3 on the next request.

5
Live site updated

Within minutes, the changes are live at https://resq.com.ng for visitors worldwide.

LESSONS LEARNED

This project pushed beyond basic cloud concepts and into real-world infrastructure challenges. Some of the key things learned along the way:

  • ACM certificates for CloudFront must always be created in us-east-1 regardless of where your other resources are.
  • Importing existing AWS resources into Terraform requires careful state management — each resource needs to be imported before applying changes.
  • CloudFront distributions with WAF or pricing plan subscriptions have restrictions on what can be changed, requiring the Terraform config to match the existing setup exactly.
  • The principle of least privilege matters — the GitHub Actions IAM user only has S3 and CloudFront invalidation permissions, nothing more.
  • DNS propagation for .com.ng domains can be slower than standard .com domains due to the .ng registry.
  • Infrastructure as Code makes the entire setup reproducible — the whole project can be rebuilt from scratch by running terraform apply.
